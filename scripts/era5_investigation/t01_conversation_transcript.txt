## ERA5 Training Data Analysis - Key Findings

### 1. specific_total_water_1 (97.7 hPa) - Extreme non-Gaussian distribution

This is the single most striking finding. **Skewness = 190, kurtosis = 124,000**.

- **95.4% of variance** comes from the top 1% of values
- Background (bottom 95%): mean=2.8e-6, std=4.7e-7 (narrow, well-behaved)
- But extreme values reach 3.8e-3 (**1000x the mean**), caused by deep convective overshooting into the lower stratosphere
- Extremes are concentrated in the tropics (10S-20N)
- Temporal autocorrelation only **0.80** (vs 0.99+ for most variables)
- The full/residual scaling ratio is **0.86** (residuals are *larger* than the full-field std!)
- The residual kurtosis is **130,290** - extreme outliers in the timestep-to-timestep changes

**Implication**: The normalization scale is dominated by rare tropical convective events. For 99% of data points, the normalized signal is tiny. A log transform or mixture model might be needed for this variable. The loss gradient is probably dominated by these rare events.

### 2. northward_wind_0 (25.6 hPa) - Nearly unpredictable at 6h resolution

- Global-mean temporal autocorrelation at lag-1 (6h): **0.37** (essentially decorrelated)
- At the poles: autocorrelation **0.024** (pure noise)
- Kurtosis = **10.79** (very heavy tails)
- Contrast: eastward_wind_0 at the same level has autocorrelation **0.999** (highly predictable)

The difference is physical: the stratospheric circulation (QBO, polar vortex) is predominantly **zonal**, so the meridional wind at this level has very little coherent temporal structure. The model is essentially being asked to predict noise.

**Implication**: Loss on northward_wind_0 may be adding noisy gradients without meaningful learning signal. Consider reducing its loss weight or removing it from outputs.

### 3. specific_total_water_0 (25.6 hPa) - Nearly constant field

- Coefficient of variation: only **7%**
- Range: [1.85e-6, 3.85e-6] - dynamic range of just 2.1x
- 6h changes are only **8.8% of the full-field std** in normalized space
- Residual std = 2.1e-8 (normalization scale 2.4e-7 → 12x ratio)

This is stratospheric water vapor, which changes very slowly. The model needs extreme precision to predict meaningful changes.

**Implication**: This variable is well-handled by the full-field normalization since the full-field signal is small and the model just needs to persist it accurately.

### 4. ERA5 data discontinuities affect training data

The training config skips some problematic periods but **includes 1979-1995**, which spans two known discontinuities:
- **1986**: specific_total_water_0 jumps **-3.9σ** (parallel stream merge)
- **1993**: air_temperature_0 jumps **-3.1σ**, air_temperature_1 jumps **-2.5σ**

The stratospheric variables (levels 0-1) are most affected by these reanalysis artifacts.

### 5. Radiation variables are bimodal and zero-inflated

- **DSWRFsfc**, **USWRFtoa**: exactly zero **29%** of the time (nighttime)
- **USWRFsfc**: 23% negative values (tiny, ~-0.00002 W/m2 - float precision noise)
- Residual std **exceeds** full-field std for DSWRFsfc (ratio 0.78) because the diurnal cycle creates transitions of ~1000 W/m2
- The model config correctly forces positive values, but the bimodal distribution means standard Gaussian normalization is poorly suited

### 6. PRATEsfc - Extreme precipitation distribution

- **18% zeros**, skewness **9.87**, kurtosis **198**
- Residual kurtosis **166** - extreme outliers in precipitation changes
- The non-zero distribution spans 6 orders of magnitude (6e-9 to 4e-4 at P1-P99)

1. **specific_total_water_1**: Consider log-transform, mixture-of-experts, or separate handling of background vs extreme convective events

### 2. **Loss Function Architecture (EnsembleLoss with CRPS and Energy Score)**

**Location:** `/home/mcgibbon/python/ace/fme/core/loss.py`

**EnsembleLoss class (lines 303-342):**
- Combines CRPS (Continuous Ranked Probability Score) and Energy Score with configurable weights
- Takes `crps_weight` and `energy_score_weight` parameters
- Uses `alpha=0.95` for "almost-fair" CRPS modification (from https://arxiv.org/html/2412.15832v1)

**CRPS Loss (lines 286-300):**
- Implements: `E[|X - y|] - (1-epsilon) * (1/2) * E[|X - X'|]`
- Takes ensemble predictions `[batch, ensemble, ...]` and target `[batch, 1, ...]`
- Computes mean absolute difference between ensemble and target, minus pairwise ensemble spread

**Energy Score Loss (lines 238-283):**
- Computed in spectral domain via Spherical Harmonic Transform (SHT)
- Formula: `E[||X - y||] - 0.5 * E[||X - X'||]` for complex spectral coefficients
- Scaled by `2 * sqrt(n_l * n_m)` to match CRPS magnitude

**Loss Application Flow (lines 58-74 in WeightedMappingLoss):**
1. Normalize predictions and targets using loss normalizer
2. Pack variable dictionaries into tensors by channel dimension
3. Handle NaNs by masking them to zero
4. Apply variable-specific weights (different for each variable/level)
5. Pass to loss function

```yaml
- specific_total_water_0 through specific_total_water_7
- eastward_wind_0 through eastward_wind_7
- northward_wind_0 through northward_wind_7
```

**Key Point:** Level 0 is the top of atmosphere (highest altitude), level 7 is lowest. These are treated as **separate channels** in the network:
- Each level is a distinct input/output channel
- They have independent normalization parameters (separate mean/std in centering.nc and scaling files)
- They have independent loss weights (line 72-75 in config: different weights for `air_temperature_0` vs `air_temperature_1`)

**Loss Weighting:**
Different levels can have different loss weights:
```yaml
weights:
  air_temperature_0: 0.5
  air_temperature_1: 0.5
  specific_total_water_0: 0.5
  specific_total_water_1: 0.25  # Lower weight for level 1 water
  specific_total_water_2: 0.5
```

### 5. **Residual Prediction Support**

**Location:** Lines 544-545 in `/fme/core/step/single_module.py`

When `residual_prediction=true`:
- Network predicts **tendencies** (changes) instead of full fields
- Output is added to input: `output = input_normalized + network_output`
- Loss is still computed but against residuals, using `scaling-residual.nc` statistics

This is configured through `residual_normalization` in the stepper config (lines 130, 324-328 of normalizer.py).
