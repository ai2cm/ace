> ## 2. Key Technical Concepts
> 
> - **W_vs2 pathway**: New bilinear operation in `VectorDiscoConvS2` — applies the DISCO bearing-gradient filter (cos_φ/sin_φ weighted contractions `f_cp`, `f_sp`) to scalar input, then takes pointwise dot product with local vector input. Produces scalar output. Requires `N_s_in == N_s_out` (diagonal constraint: output channel o = input scalar channel c).
> - **DISCO gradient convention**: W_sv with `d=1` (index into the 2 direction dimension) gives the DISCO gradient, specifically gradient `(-f_sp/R, f_cp/R)`. Therefore V·∇s = `(v*f_cp - u*f_sp)/R` — this is W_vs2's `d=1` direction, NOT `d=0`.
> - **W_vs2 direction dimensions**:
>   - `d=0`: `u*f_cp + v*f_sp` (co-bearing dot product)
>   - `d=1`: `v*f_cp - u*f_sp` (bearing-aligned dot product = V·∇s)
> - **Diagonal constraint**: Weight shape `(N_s, N_v_in, K_v, 2)` instead of `(N_s_out, N_s_in, N_v_in, K_v, 2)` — each output scalar channel o only receives from input scalar channel c=o. Reduces parameter count by factor N_s.
> - **adv_conv design**: Dedicated `VectorDiscoConvS2(2K, K, 2K, 0, ...)` with only W_vs2 active (W_ss and W_vs zeroed). Not wrapped in VectorDiscoBlock (no residual, no MLP).
> - **Vertical coordinate hierarchy**: ShallowWater → isobaric Primitive → Sigma (p_s prognostic) → Hybrid (a_k + b_k·p_s general).

> ## 5. Problem Solving
> 
> **Why W_vs2 can't be a simple linear weight**: V·∇s is bilinear (product of V and ∇s), not linear in a single input. The operation requires: (1) convolve scalar with bearing filter → gradient, (2) pointwise dot with local V. This two-step process is what W_vs2 implements, reusing `f_cp`/`f_sp` intermediates already computed for W_sv.
> 
> **Why diagonal constraint works**: For physical advection, each T_k is advected only by V_k at the same level — no cross-level mixing. Reduces shape from `(N_s_out, N_s_in, N_v_in, K_v, 2)` to `(N_s, N_v_in, K_v, 2)`.

`ShallowWaterStepper` is a linearized, single-layer model with no thermodynamics. It's a separate lineage, not a special case of the others.

`HybridCoordinateStepper` is the most general. Setting a_k=0 recovers `SigmaCoordinateStepper`. Further fixing p_s=const eliminates vertical advection and recovers the physics of `PrimitiveEquationsStepper`.

### "Latest" and Missing Features

The most complete physics reference is `HybridCoordinateStepper`. `PrimitiveEquationsBlockStepper` is the "latest" in the sense of being the block-architecture design target, but it's currently limited to isobaric (fixed-pressure-level) physics, equivalent to `PrimitiveEquationsStepper`.

**Features present in `HybridCoordinateStepper` but absent from `PrimitiveEquationsBlockStepper`:**

1. **Prognostic surface pressure** (p_s) — no continuity equation, mass is not conserved
2. **Vertical advection** — no ω·∂/∂p terms for V, T, q
3. **General vertical coordinates** — fixed pressure levels only; no sigma or hybrid-sigma levels
4. **Divergence-driven pressure tendency** — ∂p_s/∂t = −∫∇·(V dp) is absent
5. **Pressure-weighted vertical integral** — needed for correct omega (ω) computation in non-isobaric coords

In short, `PrimitiveEquationsBlockStepper` encodes the correct horizontal dynamics (PGF, Coriolis, horizontal advection of T/q via W_vs2) but is missing the vertical dimension of the full primitive equations that `HybridCoordinateStepper` has.

**What's reducible vs. irreducible:**

The remaining explicit computations fall into three categories:

1. **Genuinely state-dependent coefficients — can't go in a fixed-weight block:**
   - **φ recurrence**: `φ_k = φ_{k-1} + R T_{k-1} log(p_{k-1}/p_k)` — the log ratio `log(p_{k-1}/p_k)` depends on `p_s(x,y)` (since `p_k = a_k + b_k p_s`), so it's spatially varying. In the isobaric case this is a fixed constant and CAN be encoded as a lower-triangular W_sv. In hybrid it cannot without approximation.
   - **PGF correction**: `−(RT_k b_k/p_k) ∇p_s` — trilinear in T, p_s, ∇p_s.
   - **dp_s/dt weights**: `dp_k = Δa_k + Δb_k p_s` multiplies divergence — bilinear.
   - **C_k recurrence**: sequential and state-dependent.

2. **Vertical operations — spatial DISCO doesn't apply:**
   - `∂X/∂p` (finite differences in the pressure/vertical direction)
   - Vertical advection `-C_k ∂X/∂p`
   - `ω_k = b_k(∂p_s/∂t + V·∇p_s) + C_k` (pointwise assembly)

3. **Quadratic in velocity — no linear filter:**
   - `KE = 0.5(u² + v²)` — must be precomputed before block call.

**The one thing that approximation could handle:** If we accept replacing `log(p_{k-1}/p_k)(x,y)` with a fixed reference value (same as what the isobaric block stepper does), the φ recurrence becomes a lower-triangular T → -∇φ coupling in W_sv, and we could eliminate the explicit φ computation. But that's an approximation, not exact hybrid-coordinate physics.

> Let’s talk for a moment about the vertical recurrent relationships. Is there any way to approximate solutions to this type of equation with some kind of fancy matrix inversion or other iterative solution?

The vertical recurrences in this model are all **lower-triangular forward sweeps**, not systems requiring inversion. Let me break them down:

---

## The Three Recurrences

### 1. Hydrostatic geopotential (`_hydrostatic`)
```
φ_k = φ_surface + ∑_{j=0}^{k-1} R T_j ln(p_j / p_{j+1})
```
This is a **cumulative sum** of `R T_j * log_ratio_j`. The loop can be eliminated entirely:
```python
increments = self.R * T[:, :-1] * log_ratio   # (B, K-1, H, W)
phi[:, 1:] = self.phi_surface + torch.cumsum(increments, dim=1)
```

### 2. C interfaces (`_C_interfaces`)
```
C_{k+1/2} = C_{k-1/2} + forcing_k
```
Same structure — a cumsum of per-level forcing:
```python
forcing = -db * dp_s_dt - dp_k * div - db * v_dot_gps   # (B, K, H, W)
C_half = torch.cat([zeros, torch.cumsum(forcing, dim=1)], dim=1)
```

### 3. Vertical finite differences (`_dX_dp`)
This is multiplication by a **tridiagonal difference matrix** and can be expressed as a single matmul over the level dimension.

---

## Why No Inversion Is Needed

All three are strictly **lower-triangular** (each level k depends only on levels 0..k-1). Forward substitution is exact — there's nothing to invert. The loops are just an inefficient way to compute cumulative sums.

The `torch.cumsum` replacements are the "fancy" version: they run in O(log K) steps in parallel on GPU via a parallel prefix scan, vs O(K) sequential steps in the loop.

---

## Where Inversion Would Appear

Inversion only enters if you want **implicit time stepping** — e.g., treating the acoustic/gravity-wave terms implicitly to allow larger dt. That gives a tridiagonal system per column (from the vertical structure), solvable by the Thomas algorithm (O(K)). This is what operational NWP models do (semi-implicit schemes). The current explicit RK4 doesn't need it.

---

## Practical Implication for the Block Stepper

The `_hydrostatic` loop is currently the main bottleneck inside `compute_tendencies` that isn't vectorized. Replacing it with `torch.cumsum` would be a clean win, especially for large K. Want me to do that replacement?

> Ok, based on that is it possible to encode these relationships by carefully initializing weights in a MLP? It may need to be non residual i suppose.

Yes, and the answer is quite clean.

## The Isobaric Case: Pure Linear Layers

All three vertical operations are **linear maps over the level dimension**. Each can be encoded exactly as a `nn.Conv2d(K, K_out, 1, bias=False)` — a pointwise 1×1 conv is just a learned linear mix of channels at each spatial point independently.

### Hydrostatic φ
The recurrence:
```
φ_k = φ_surface + ∑_{j<k} R T_j log(p_j/p_{j+1})
```
is exactly a lower-triangular matrix–vector product over levels:
```
W_hydro[k, j] = R * log(p_j/p_{j+1})   for j < k
              = 0                         for j >= k
```
One frozen `Conv2d(K, K, 1)` with this weight matrix, applied to T, gives all K geopotentials simultaneously. No loop.

### C interfaces
```
C_{k+1/2} = ∑_{j≤k} forcing_j
```
This is a lower-triangular all-ones matrix (scaled by Δb, Δp) times the forcing vector. One more frozen `Conv2d(K, K+1, 1)`.

### ∂X/∂p (finite differences)
A tridiagonal matrix with central-difference coefficients `1/(p_{k+1} - p_{k-1})`. One frozen `Conv2d(K, K, 1)`.

---

## You're Right: Non-Residual

The residual structure `y = x + f(x)` is wrong here — φ is a new quantity computed from T, not a correction to T. These should be plain `y = W @ x` linear layers with no skip connection and no activation.

---

## The Hybrid Complication

In hybrid coordinates, the hydrostatic log_ratio:
```
ln((a_j + b_j p_s) / (a_{j+1} + b_{j+1} p_s))
```
depends on p_s, making the computation **bilinear in (T, g(p_s))**. A fixed linear layer can't capture this — it would need a multiplicative pathway: one branch processes T, another processes p_s, and the result is multiplied before the cumsum. This is exactly the same structure as W_vs2 (which multiplied the gradient of a scalar with a vector). A "vertical W_vs2" analog could encode hybrid hydrostatics.

---

## The Bigger Picture

This is the same idea as the horizontal block — encode physical operators as frozen structured weight matrices, then optionally unfreeze to learn corrections. The vertical direction would use `Conv2d(K, K, 1)` (pointwise in space, mixing in level) instead of DISCO (spatial mixing, diagonal in level). The two form a complete basis:

| Layer type | Mixes | Diagonal in |
|---|---|---|
| DISCO (W_sv, W_vs, W_vs2) | Spatial neighbors | Level |
| Vertical Conv2d(K, K, 1) | Levels | Spatial |

Together they span both directions. Want me to replace the for loops with these frozen linear layers?

> The W_vs has divergence and curl parts, if we added a third index for magnitude could we use it to compute kinetic energy and avoid the precomputation?

No — and the reason is fundamental: W_vs is **linear** in V, but KE = ½(u² + v²) is **quadratic**.

---

## Why a Third Direction Index Doesn't Work

The two existing W_vs directions are linear projections onto the bearing and perpendicular axes:
- `d=0`: `u·f_cp + v·f_sp` (bearing component)
- `d=1`: `v·f_cp - u·f_sp` (perpendicular component)

Adding a `d=2` "magnitude" direction would require computing `sqrt(u² + v²)` or `u² + v²` inside the filter — neither is linear in V. No linear combination of filter weights can produce a quadratic expression.

---

## What Could Work: a New `VectorDotProduct` Pathway

KE is the pointwise dot product `V · V` — no spatial filtering, just a local quadratic product. This could be a new module:

```
V_c → ½ (u_c² + v_c²)     [or a learned mixing: ∑_v w_{c,v}(u_v² + v_v²)]
```

It's structurally like ScalarVectorProduct but reversed — "V modulates a scalar" instead of "s modulates a vector." Initialized with `w_{k,k} = 0.5`, it would remove the KE precomputation and allow the block to receive raw V and compute KE internally.

---

## Comparison

| Pathway | Bilinear in | Can produce |
|---|---|---|
| W_vs (`d=0,1`) | (filter_basis, V) | curl, div — linear in V |
| W_vs2 | (filter_basis·s, V) | V·∇s — bilinear in (s, V) |
| **VectorDotProduct** (new) | **(V, V)** | **|V|² = KE·2 — quadratic in V** |

The precomputation `ke = 0.5*(u²+v²)` is doing exactly what this hypothetical module would do. It's cheap (O(K·H·W), no spatial filtering needed), so encoding it in a module is primarily an architectural cleanness choice, not a performance one.
