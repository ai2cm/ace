**Why the product rule doesn't work for DISCO operators:**

The DISCO divergence computes:
```
div(sV)_i = Σ_j ψ(i,j) · [cos_β(i,j) · s_j · u_j + sin_β(i,j) · s_j · v_j]
```

The scalar field `s_j` appears at the **source point j** inside the convolution sum. The product rule `V·∇s = div(sV) − s·div(V)` would require factoring `s_i` out at the **output point i**, which is a pointwise operation incompatible with any finite-support convolutional operator. This is not a smoothness or approximation issue — it's a fundamental structural property of convolutions.

Momentum advection sidesteps this entirely via the vorticity form `ζ×V − ∇KE`, which replaces `V·∇V` with curl→scalar-vector product and KE gradient — both operations native to the block. No analogous reformulation exists for `V·∇T`.

**Solution:** `HorizontalAdvection` computes `-V·∇T` directly via the DISCO gradient operator (W_sv) + pointwise dot product, batching all K×n_tracers fields in a single conv forward pass.

**What W_vs actually computes:**

The "2" dimension (index `d`) is not "each component of the vector (u, v)" — it's **{curl, div}**, i.e., the two scalar invariants formed by projecting V onto the bearing direction β from source j to output i:

```
div  = Σ_j ψ_k(i,j) · [cos_β(i,j)·u_j + sin_β(i,j)·v_j]   # radial projection
curl = Σ_j ψ_k(i,j) · [−sin_β(i,j)·u_j + cos_β(i,j)·v_j]  # tangential projection
```

Then `W_vs[o, c, k, 0]` weights the curl and `W_vs[o, c, k, 1]` weights the div. The scalar output is a linear combination of these two convolutional divergence/curl operators.

**How it's used in the block:**

Yes, W_vs is live and critical:
- `W_vs[_ch(k, _ZETA_KIND), k, :, 0] = 1/R` — extracts vorticity ζ_k (curl of V_k)
- `W_vs[_ch(k, _DIV_KIND),  k, :, 1] = 1/R` — extracts divergence δ_k (div of V_k)

ζ_k then feeds the `ScalarVectorProduct` for vorticity advection + Coriolis, and δ_k feeds the ω computation.

**Why this still can't compute V·∇T:**

The W_vs operation is a convolution over the neighborhood of each output point. It can compute any linear combination of div(V) and curl(V), but V·∇T requires multiplying V_i (the velocity *at output point i*) by ∇T_i (the gradient *at output point i*). That's a pointwise product of two quantities both evaluated at the same point — something no convolutional weight can express, because any weight matrix acts across neighborhoods, not at a single point. This is the same fundamental reason the product rule failed.
