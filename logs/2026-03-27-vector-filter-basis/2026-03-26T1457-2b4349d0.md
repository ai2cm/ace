Great question. After reading the document and studying the DISCO implementation, the answer is **yes** — the same FFT-based longitude-invariance optimization fully applies to the directional filter basis.

## Why the optimization still works

The current optimization exploits the fact that the filter tensor `psi(lat_out, lat_in, lon_in - lon_out)` depends only on the longitude **difference**, not absolute longitude. This makes the convolution along longitude a circular cross-correlation, computable efficiently via FFT.

The directional filters `R(r) · W_m(r) · sin(mφ)` and `R(r) · W_m(r) · cos(mφ)` are defined in terms of:
- `r` = geodesic distance from output point to input point
- `φ` = azimuthal angle from output point to input point

Both `r` and `φ` depend only on `(lat_out, lat_in, lon_in - lon_out)`. They do **not** depend on absolute longitude. So each directional filter basis function has exactly the same longitude-translation invariance as the current isotropic basis, and the banded-FFT pipeline in `_disco_utils.py` applies without modification.

## How the cube reference frame fits in

The cube reference frame defines an orientation angle `α(lat, lon)` at each output point. The actual filter used at output point `(lat, lon)` measures azimuth relative to this reference direction, giving `φ - α`. Expanding:

```
sin(m(φ - α)) = sin(mφ)·cos(mα) - cos(mφ)·sin(mα)
cos(m(φ - α)) = cos(mφ)·cos(mα) + sin(mφ)·sin(mα)
```

This means the cube-frame-corrected output is a **pointwise linear combination** of the "raw" convolution outputs for `sin(mφ)` and `cos(mφ)`:

```
output_sin(lat, lon) = cos(mα(lat,lon)) · conv_sin(lat,lon) - sin(mα(lat,lon)) · conv_cos(lat,lon)
output_cos(lat, lon) = sin(mα(lat,lon)) · conv_sin(lat,lon) + cos(mα(lat,lon)) · conv_cos(lat,lon)
```

where `conv_sin` and `conv_cos` are computed using the existing FFT pipeline (each is a standard DISCO convolution with a longitude-invariant filter).

## Implementation strategy

1. **Filter precomputation**: For each angular mode `m` and radial scale, precompute two banded FFT filter tensors (sin and cos) — same as the current `_precompute_psi_banded()`, just with `sin(mφ)` and `cos(mφ)` multiplied into the filter values.

2. **Forward pass**: Run the existing `_disco_s2_contraction_fft()` once for each filter basis function (2 per angular mode, 1 for `m=0`). This is the expensive step, and it uses the full FFT optimization.

3. **Reference frame correction**: Precompute `α(lat, lon)` from the cube projection (once, at init). After convolution, apply the 2×2 rotation per `(sin, cos)` pair as a cheap pointwise multiply — `O(nlat × nlon)` per mode, negligible compared to the convolution itself.

## Cost relative to the isotropic case

The additional cost is linear in the number of filter basis functions:
- `m = 0`: 1 filter (isotropic, same as now)
- Each `m ≥ 1`: 2 filters (sin + cos pair)
- Total for `M` angular modes: `1 + 2M` filters vs. the current `N_radial` isotropic filters

The per-filter cost is identical. The cube reference frame correction and Z₄ transition maps at vertices are pointwise operations with negligible cost. So the fundamental performance character of DISCO convolution is preserved — no new algorithmic bottleneck is introduced.

Not exactly — let me clarify. The azimuthal angle `φ` in the filter is **not** measured relative to north. It's the azimuthal angle from the output point to the input point in the rotated coordinate system where the output point is at the north pole. This angle has nothing to do with any reference frame convention — it's purely geometric.

The current isotropic filter ignores `φ` entirely. The directional filters use `sin(mφ)` and `cos(mφ)` where `φ` is this geometric azimuthal angle. These are already well-defined and longitude-invariant without any reference frame.

The cube reference frame only enters when you want to **interpret** the output channels. The raw convolution with `sin(mφ)` picks out a directional feature aligned with a particular geometric direction (roughly "north" at the output point, since `φ=0` points along the meridian in the Euler angle rotation). The raw `cos(mφ)` convolution picks out the orthogonal direction.

So what I'm actually saying is:

1. **Compute convolutions** with `sin(mφ)` and `cos(mφ)` filters using the existing FFT pipeline. These filters are defined purely in terms of the geometric relationship between output and input points — no reference frame needed. They enjoy full longitude invariance.

2. **Post-convolution**, apply a pointwise 2x2 rotation to re-express the (sin, cos) pair from the "Euler angle frame" (where `φ=0` is along the meridian) into the cube reference frame. This is where `α(lat, lon)` enters — it's the angle between the meridian direction and the cube frame direction at each output point.

The reason you'd want step 2 at all is for the equivariance properties described in the document — the cube frame ensures the filter basis is closed under the Z₄ transition maps at vertices, which matters for how you apply nonlinearities between layers.

**But here's the thing** — step 2 is optional from a computational standpoint. You could skip the cube frame entirely and just let the network learn in the "meridian-aligned" frame. The problem is that this frame is singular at the poles (all meridians converge), so `sin(mφ)` and `cos(mφ)` become ill-defined there — not because the filter values are wrong, but because the directional meaning of the output channels becomes degenerate.

This is where your suggestion about polar caps becomes relevant. A practical alternative to the full cube construction:

- **Away from poles**: Use the meridian-aligned frame directly. `sin(mφ)` and `cos(mφ)` have clear directional meaning, the FFT optimization works, and no reference frame machinery is needed.
- **Near poles**: The meridian frame degenerates, so you need *some* treatment. Options include:
  - Multiplying by `cos^m(lat)` to smoothly suppress directional modes (the document mentions this as the "naive" approach but notes it kills sensitivity at high latitudes)
  - Switching to a rotated coordinate system for polar caps (e.g., rotate the sphere 90° and recompute on the cap)
  - Using the cube frame only in the polar regions

The cube construction in the document is the "do it right everywhere" approach. But if you're willing to accept a simpler scheme that handles poles specially, you could avoid the cube frame entirely and still get the FFT optimization for all latitudes.

Does this distinction make sense? The core point is: the `sin(mφ)` and `cos(mφ)` convolutions are well-defined and efficient regardless of reference frame — the reference frame question is about how you interpret and use the outputs between layers.

The document is written at `vector_filter_basis.md`. Here's a summary of the key points:

**Core idea**: Keep isotropic radial filters, but bake the frame-rotation matrix `R(γ)` into the filter tensor for vector channels. The rotation angle `γ = φ - β` is computable from the existing Euler rotation geometry (a few extra dot products per support point in the precomputation loop).

**Three filter tensors**: `psi_scalar` (unchanged), `psi_cos = ψ·cos(γ)`, and `psi_sin = ψ·sin(γ)`. All three are longitude-invariant, so the FFT optimization applies to each.

**Cost**: Vector channels cost 2x more than if treated as scalars (4 contraction calls per vector pair vs 2), because the rotation varies per input point and can't be factored out of the sum. This is the fundamental difference from the cube approach (where the rotation is at the output point and costs nothing).

**Key design decisions documented**:
- Weight tensor is unconstrained — freely mixes scalar/u/v intermediates since they're all in the output frame
- Type labels are bookkeeping for the next layer, not a constraint on mixing
- Cross-type interactions (scalar x vector -> vector) happen in nonlinearities, not convolution
- Safe vs unsafe nonlinearities for maintaining vector structure
