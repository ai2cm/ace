## Why `∇(constant) ≠ 0` in the DISCO gradient operator

### What the operator computes

The scalar→vector path uses bearing-modulated filter kernels. For each output point `t` the gradient approximation is:

```
output_u(t) = Σ_{(i,j) ∈ support(t)}  ψ_k(r_{tij}) · sin(φ_{tij}) · f(i,j) · w_i
output_v(t) = Σ_{(i,j) ∈ support(t)}  ψ_k(r_{tij}) · cos(φ_{tij}) · f(i,j) · w_i
```

where `r_{tij}` is the angular distance from `t` to input point `(i,j)`, `φ_{tij}` is the bearing angle, and `w_i` are the Gauss–Legendre quadrature weights (latitude-only).

### Why it should be zero in theory

For a constant input `f = 1`, the output reduces to:

```
Σ_{(i,j)} ψ_k(r) · cos(φ) · w_i
```

On a continuous sphere this integral is **exactly zero**: the integrand `ψ_k(r)·cos(φ)` is antisymmetric under `φ → φ+π` (reflection through the center), and the spherical cap has that symmetry.

### Why the discrete version breaks this

Two effects break the antisymmetry:

1. **Hard filter boundary creates unmatched pairs.** The support is the discrete set of grid points with `r < θ_cutoff`. A grid point at bearing `φ` has a "mirror" partner at bearing `φ+π`. These two are at different angular distances from the output point — they are symmetric only if the grid is perfectly uniform in all directions. Near the filter edge, one partner can fall just inside the cutoff while the other falls just outside, so `cos(φ)` is added but `cos(φ+π)` is missing. The sum doesn't cancel.

2. **Quadrature weights `w_i` depend only on latitude, not longitude.** The pair at bearing `φ` and `φ+π` are at the same latitude only if they are due east/west of the output point. In general they sit at different latitudes and receive different weights, so even when both fall inside the support the weighted contributions don't cancel exactly.

The net error is proportional to the fraction of "split pairs" at the support boundary, which is roughly `Δgrid / θ_cutoff`. For `(16×32)` grid with `θ_cutoff = 2π/(nlat−1) ≈ 0.42 rad`, this fraction is order `1/16 ≈ 6%` — consistent with the ~1% spurious response observed in the tests (the filter kernel also tapers toward zero near the boundary, reducing the contribution of the split pairs).

### How to fix it properly

There are three approaches, from most to least invasive:

**Option A – Zero-mean correction during kernel precomputation (cleanest long-term fix)**  
After computing `psi_vals · cos(φ)` and `psi_vals · sin(φ)` for each output latitude `t`, subtract the weighted mean from each kernel–latitude slab so that the quadrature sum is exactly zero:

```python
# In _precompute_vector_convolution_tensor_s2, after computing vals_cp:
for k in range(kernel_size):
    for t in range(nlat_out):
        # indices in this (k, t) slab
        mask = (out_idx[0] == k) & (out_idx[1] == t)
        w = quad_weights[out_idx[2][mask] // nlon_in]
        vals_cp[mask] -= (vals_cp[mask] * w).sum() / w.sum()
        vals_sp[mask] -= (vals_sp[mask] * w).sum() / w.sum()
```

This enforces `Σ ψ·cos(φ)·w = 0` exactly at every output point and makes `∇(const) = 0` in the discrete operator, at no additional runtime cost.

**Option B – Spatial mean subtraction at call time (current approach)**  
Before applying the differential operator, subtract the spatial mean of the input field. This is exact: `∇(f − mean(f)) = ∇f` in exact arithmetic, and `mean(f) = const` has zero gradient. This is what we do for `φ`, `T_anom`, and `q_anom` in `PrimitiveEquationsStepper`. The cost is one global reduction per forward pass and one extra convolution.

**Option C – Larger filter support / finer grid**  
Increasing `θ_cutoff` (or using a higher-resolution grid) reduces the relative weight of the support boundary. The error scales as `O(Δgrid / θ_cutoff)`, so doubling the resolution or the support both halve the error. This is the most expensive option and never reaches exactly zero.

**Recommendation:** Option A is the right principled fix — it guarantees `∇(const) = 0` in the discrete kernel itself and requires no changes to the callers. Option B is a perfectly adequate workaround at the application layer, and is what we have now.
